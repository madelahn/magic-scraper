## Spacing Scale

Declared values (must be multiples of 4). Source: existing `/games` page pattern + Tailwind v4 conventions.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline badge padding (`gap-1`, `p-1`) |
| sm | 8px | Compact element spacing, label-to-value gaps (`gap-2`, `p-2`) |
| md | 16px | Default element spacing, card internal padding (`px-4`, `p-4`) |
| lg | 24px | Section padding, page vertical rhythm (`py-6`, `gap-6`) |
| xl | 32px | Layout gaps between chart cards (`gap-8`, `mb-8`) |
| 2xl | 48px | Major section breaks between dashboard sections (`mb-12`) |
| 3xl | 64px | Page-level top/bottom padding (not used at this scale in existing pages) |

Exceptions:
- Touch target for mobile expand/collapse button: minimum 44px height (`min-h-[44px]`) per accessibility standard
- Chart container heights are declared per chart type (see Chart Containers section below)

| Role | Size | Weight | Line Height | Tailwind Class | Usage |

## Chart Chrome (Adaptive to Light/Dark)

Non-data chart elements adapt to mode via Tailwind CSS variable resolution. Because Recharts renders SVG and cannot natively consume Tailwind utility classes on SVG attributes, chart chrome reads the CSS variable value at render time using this pattern:

```ts
// Read at component render time (inside "use client" component)
const style = typeof window !== 'undefined'
  ? getComputedStyle(document.documentElement)
  : null;

const chartTokens = {
  foreground: style?.getPropertyValue('--foreground').trim() ?? '#18181b',
  muted:      style?.getPropertyValue('--muted').trim()      ?? '#71717a',
  border:     style?.getPropertyValue('--border').trim()      ?? '#e4e4e7',
  surface:    style?.getPropertyValue('--surface').trim()     ?? '#f4f4f5',
};
```

| Chrome Element | Recharts Prop | Token Used | Example Value (light) |
|----------------|---------------|------------|----------------------|
| Axis lines | `axisLine={{ stroke: chartTokens.border }}` | `--border` | `#e4e4e7` |
| Axis tick labels | `tick={{ fill: chartTokens.muted, fontSize: 12 }}` | `--muted` | `#71717a` |
| Cartesian grid lines | `stroke={chartTokens.border}` | `--border` | `#e4e4e7` |
| Tooltip background | inline style `background: chartTokens.surface` | `--surface` | `#f4f4f5` |
| Tooltip border | inline style `border: 1px solid chartTokens.border` | `--border` | `#e4e4e7` |
| Tooltip text | inline style `color: chartTokens.foreground` | `--foreground` | `#18181b` |
| Legend text | `formatter={(v) => <span style={{color: chartTokens.muted}}>{v}</span>}` | `--muted` | `#71717a` |
| Polar grid (radar) | `gridType="polygon"` stroke → `chartTokens.border` | `--border` | `#e4e4e7` |

The `chartTokens` object is computed once per page mount (not per chart render) and passed as a prop or shared via a module-level reference within the client bundle. The ThemeToggle in `src/app/components/theme-toggle.tsx` toggles `.dark` on `<html>`, which updates the CSS variables; however, chart tokens will NOT auto-recompute on theme toggle without a re-render. Acceptable for Phase 7 — if the user toggles theme, the stats page will reflect new tokens on the next navigation/mount. A later phase can add a theme-change listener if needed.

## Responsive Behavior

### Breakpoint

`sm:` (640px) — matches existing `header.tsx` and `games/page.tsx` convention (D-07).

### Desktop (sm: and above)

- All charts render unconditionally (`sm:block` always visible on desktop).
- Each chart inside `<ResponsiveContainer width="100%" height={N}>` fills the card width.
- Breakdowns section: `sm:grid sm:grid-cols-2 sm:gap-6` places the two pie charts side-by-side.
- All other sections: single column full-width.

### Mobile (below sm:)

Each chart section renders as a collapsed summary card by default. State: `useState<Set<string>>` named `expandedCharts`, tracking which chart IDs are expanded.

**Chart IDs (string constants):**

```ts
const CHART_IDS = {
  RADAR:          'player-radar',
  PLAYER_WIN_BAR: 'player-win-rate',
  DECK_WIN_BAR:   'deck-win-rate',
  WINS_BY_PLAYER_PIE: 'wins-by-player',
  GAMES_BY_DECK_PIE:  'games-by-deck',
  WEEKLY_FREQ:    'weekly-frequency',
  LIKELY_BUMP:    'most-likely-bump',
} as const;
```

**Mobile card anatomy:**

```html
<div class="rounded-lg border border-border bg-surface sm:hidden">
  <button
    class="w-full flex items-center justify-between px-4 min-h-[44px] text-left"
    onClick="toggle(id)"
    aria-expanded="{true|false}"
  >
    <span class="text-sm font-medium text-foreground">{Chart Title}</span>
    <span class="text-xs text-muted">{Summary Text}</span>
    <ChevronDown class="w-4 h-4 text-muted transition-transform {rotated if expanded}" />
  </button>
  {expanded && (
    <div class="px-4 pb-4">
      <!-- Full Recharts chart, same as desktop -->
    </div>
  )}
</div>

<!-- Desktop always shows the card (no toggle) -->
<div class="hidden sm:block rounded-lg border border-border bg-surface p-4 mb-6">
  <!-- Full chart -->
</div>
```

**Mobile summary text per chart** (shown in collapsed state):

| Chart | Summary Text Pattern | Example |
|-------|---------------------|---------|
| Player Radar | "N players" | "12 players" |
| Player Win Rate | "Top: {name} {rate}%" | "Top: Amirali 62%" |
| Deck Win Rate | "Top: {name} {rate}%" | "Top: Yuriko 71%" |
| Wins by Player Pie | "{name} leads with N wins" | "Amirali leads with 24 wins" |
| Games by Deck Pie | "{name} most played" | "Yuriko most played" |
| Weekly Frequency | "Avg {n} games/week" | "Avg 2.3 games/week" |
| Most Likely to Play | "#1: {name} {rate}%" | "#1: Amirali 87%" |

If data is empty, summary text is "No data yet".

## Chart Containers

Heights are chosen for readability at the declared responsive contexts. Heights scale with player/deck count for bar charts.

| Chart | Component | Container Height | Notes |
|-------|-----------|-----------------|-------|
| Player Radar | RadarChart | 400px | Fixed — 4 axes, up to 15 polygons |
| Player Win Rate Bar | BarChart (horizontal) | `max(300, N * 28 + 60)` px | N = player count; 28px per bar + 60px for axes |
| Deck Win Rate Bar | BarChart (horizontal) | `max(300, N * 28 + 60)` px | N = deck count (capped at 20 bars max; see note) |
| Wins by Player Pie | PieChart | 320px | Fixed |
| Games by Deck Pie | PieChart | 320px | Fixed |
| Weekly Frequency Line | LineChart | 280px | Fixed |
| Most Likely to Play Bump | LineChart (inverted Y) | 340px | Fixed — rank lines need more vertical space |

All containers use `<ResponsiveContainer width="100%" height={N}>`. Heights are declared as integer pixel values in the component.

**Deck chart cap:** If deck count exceeds 20, render only the top 20 by win rate (or game count for the pie). No "show all" toggle in Phase 7. Add a `text-xs text-muted` note: "Showing top 20 decks" below the card when cap is applied.

## Chart-Specific Visual Contracts

### 1. Player Radar (Player Overview)

- Component: `RadarChart`
- One `<Radar>` element per player, each with `fill={color}` at `fillOpacity={0.15}` and `stroke={color}` at full opacity.
- Axes: `["Played", "Wins", "Screwed", "Won by Combo"]`
- Normalization: per-axis max (D-28). Each axis value is divided by the maximum observed value for that axis across all players before passing to Recharts. Recharts `domain={[0, 1]}` on `PolarRadiusAxis`.
- `PolarRadiusAxis` tick count: 3 (0, 0.5, 1). Labels hidden (`tick={false}`). Grid lines use `chartTokens.border`.
- `PolarAngleAxis` labels: `text-sm` font, `fill={chartTokens.foreground}`.
- `Tooltip`: custom — shows raw (unnormalized) values for the hovered player. E.g. "Played: 42, Wins: 18, Screwed: 7, Won by Combo: 5".
- `Legend`: rendered below the chart. Player names as legend entries. `iconType="circle"`.

### 2. Player Win Rate Bar (Win Rates)

- Component: `BarChart` with `layout="vertical"` (horizontal orientation)
- `XAxis type="number"` domain `[0, 1]`, `tickFormatter={(v) => \`${Math.round(v*100)}%\`}`
- `YAxis type="category"` dataKey `"player"`, width 100px, `tick={{ fontSize: 12, fill: chartTokens.muted }}`
- Long names: truncated at 14 characters with ellipsis in the YAxis tick renderer.
- Single `<Bar dataKey="rate">` with per-datum `fill` from `CHART_COLORS[index]`.
- `Tooltip`: "Player: {name} — Win rate: {rate}% ({wins}W / {played}G)"
- Sorted: highest win rate first (computed in helper, preserved in render order).
- No legend (single series, axis labels are self-explanatory).

### 3. Deck Win Rate Bar (Win Rates)

- Same structure as Player Win Rate Bar.
- `YAxis` dataKey `"deck"`, width 120px (deck names tend to be longer).
- Long names: truncated at 16 characters with ellipsis.
- `Tooltip`: "Deck: {name} — Win rate: {rate}% ({wins}W / {played}G)"
- Sorted: highest win rate first.
- No legend.

### 4. Wins by Player Pie (Breakdowns)

- Component: `PieChart` with a single `<Pie>`.
- `<Pie dataKey="wins" nameKey="player" cx="50%" cy="50%" outerRadius={120}>`
- Per-datum `fill`: `CHART_COLORS[index]`.
- Cell deprecation (Recharts 3.7+): use the `data` prop array where each entry has a `fill` field set to `CHART_COLORS[index]`. Do NOT use the deprecated `<Cell>` component.
- `Tooltip`: "{name}: {wins} wins ({pct}%)" — percentage computed from total wins in dataset.
- `Legend`: below chart, `iconType="circle"`, `formatter` wraps in `text-muted` style.

### 5. Games by Deck Pie (Breakdowns)

- Same structure as Wins by Player Pie.
- `dataKey="games" nameKey="deck"`
- `Tooltip`: "{name}: {games} games ({pct}%)"
- Imported games excluded (D-16).

### 6. Weekly Frequency Line (Frequency)

- Component: `LineChart`
- Single `<Line dataKey="gameCount" stroke={CHART_COLORS[0]}>` (steel blue `#4e79a7`).
- `XAxis dataKey="weekStart"`, `tickFormatter` abbreviates to "MMM D" format (computed without date-fns).
- `YAxis` left, integer ticks only (`allowDecimals={false}`).
- `CartesianGrid strokeDasharray="3 3"` with `stroke={chartTokens.border}`.
- `Tooltip`: "Week of {weekStart}: {gameCount} games"
- No legend (single line).
- Empty weeks (gameCount = 0) ARE included in the data array (D-22a default), which creates a flat-line visual for dry periods. If the resulting chart appears too noisy with many consecutive zeros, the executor may skip empty weeks — that is a rendering quality judgment deferred to implementation.

### 7. Most Likely to Play Bump (Frequency)

- Component: `LineChart` with `layout="horizontal"` (default).
- `XAxis dataKey="weekStart"`, same formatter as weekly frequency chart.
- `YAxis` inverted: `reversed={true}`, domain `[1, maxRank]`, integer ticks, label "Rank" (left side).
- One `<Line>` per player, `dataKey` resolved from the bump data shape (see data shape note below).
- `stroke={CHART_COLORS[index]}` per player, `dot={false}` for clean lines, `strokeWidth={2}`.
- `Tooltip`: custom — shows all player ranks for the hovered week, sorted by rank ascending.
- `Legend`: below chart, player names.

**Bump data shape note:** `computeMostLikelyToPlayBump` returns `{ weekStart: string; ranks: { player: string; rank: number }[] }[]`. Before passing to Recharts `LineChart`, pivot this into a flat array where each row is `{ weekStart, [playerName]: rank, ... }`. Recharts `<Line dataKey={playerName}>` then works natively.

**Rank tie handling:** Two players with identical participation rate through a given week share the same integer rank. Recharts lines may overlap exactly — this is acceptable for Phase 7. No visual offset or stacked marker.

## Loading and Error States

Source: `src/app/games/page.tsx` lines 100-130 pattern. Reuse verbatim.

```tsx
// Loading
{isLoading && (
  <p className="text-muted text-sm py-8 text-center">Loading stats...</p>
)}

// Error
{error && (
  <p className="text-destructive text-sm py-4">
    Failed to load stats. Try refreshing the page.
  </p>
)}
```

Loading state: displayed in place of all chart sections (no skeleton charts, no spinners). The entire `<main>` body shows the loading paragraph until `isLoading` becomes false.

Error state: displayed in place of all chart sections. No retry button in Phase 7.

## Empty State

Source: D-44, D-45.

No dedicated empty-state page. No full-page "Log your first game" CTA. Each chart renders its own "no data" treatment:

**Per-chart empty state:** When a chart's computed data array has length 0, render in place of the chart:

```html
<div class="flex items-center justify-center h-32 text-muted text-sm">
  No data yet
</div>
```

This appears inside the chart card wrapper at the same position the `<ResponsiveContainer>` would occupy. The section heading and card border remain visible.

Partial data (e.g., only imported games, which have no deck data): deck charts show "No data yet"; player charts render normally with whatever data exists (D-45).
