## Anti-pattern #1 — what actually happened, in detail

This is worth recording for next session so it doesn't recur:

**Symptom (observed by user at step 3 of the 16-step verification):**
> Going to /games spawned a "could not fetch games" error in the UI and Prisma errors in the dev server log:
> `ConnectorError { user_facing_error: None, kind: QueryError(SqliteError { extended_code: 1, message: Some("SQLITE_ERROR: no such table: main.games") }) }`
> Same error for `main.game_participants` on /api/players and /api/decks.

**Root cause:** The Phase 5 migration file `prisma/migrations/20260410_add_game_tracking/migration.sql` creates:
- `games`
- `game_participants`
- `sync_logs`

Plus 3 indexes. But it was **never run against the local dev.db**. The dev.db was created manually at some earlier point (pre-Phase 5) and only had `users` + `collection_cards`. There is no `_prisma_migrations` table, meaning Prisma migration tracking was never initialized on this DB.

**How Phase 6's executor agents didn't catch it:** The executors ran `jest` which uses a test setup (likely in-memory SQLite or a test-db) that initializes tables from the current schema on each run — so tests passed 97/97 even though the real dev.db was broken. No executor ever ran `npm run dev` and hit the real DB, so the gap was invisible until the human-verify checkpoint.

**Unblock applied this session:**
```bash
sqlite3 ./dev.db < prisma/migrations/20260410_add_game_tracking/migration.sql
```
Exit code 0. Post-state: `sqlite3 ./dev.db ".tables"` returns `collection_cards game_participants games sync_logs users`. No dev server restart required — Prisma client re-queries schema on each request in dev mode. The user should be able to refresh the browser and resume the 16-step verification from step 3.

**What's still broken (for future prevention):** Nothing forces this migration step. If someone deletes `./dev.db` and lets Prisma regenerate it, the Phase 6 tables will be missing again. Next session should land the followup work below.

## Followup work (proposed — not a blocker for 06-06 checkpoint)

A small addendum plan should be created **after** Phase 6 closes to address anti-pattern #2 and #3. Suggested scope:

### Plan 06.5 (or a Phase 7 kickoff) — "dev-onboarding migration hook"

**Must-haves:**
1. Add `"db:migrate": "prisma migrate deploy"` to `package.json` scripts.
2. Add `"predev": "npm run db:migrate"` **OR** document `npm run db:migrate` as a required onboarding step in README.
3. Change Vercel build command from default `next build` to `prisma migrate deploy && next build` (via `vercel.json` `buildCommand` field). This ensures every Vercel deployment applies pending migrations to Turso prod automatically. **Caveat:** this only works if `DATABASE_URL` in Vercel env is set to the Turso libsql URL with auth token, which it presumably already is since prod has the tables.
4. Fix `DATABASE_URL` path inconsistency: either move `dev.db` to `prisma/dev.db` and update .env, or change .env to `DATABASE_URL="file:../dev.db"`. Preference: move the DB — makes `prisma migrate status` and friends work without env hacks.
5. Init `_prisma_migrations` tracking on the existing dev.db by running `prisma migrate resolve --applied 20260117202652_init --applied 20260410_add_game_tracking` so future migrations work correctly.
6. Add a sanity test that runs `prisma migrate status` in CI (or at least in a `npm run check` script) to catch orphaned migrations before they reach dev.

**Threat model addition:** none — this is pure dev-onboarding hygiene, no new runtime surface.

**Why not fold it into Phase 6:** Phase 6's scope is game tracking feature delivery. Rolling migration-tooling fixes into it would inflate scope, delay the checkpoint, and mix concerns. Cleaner to close 06 with a note and tackle this in a focused addendum.
